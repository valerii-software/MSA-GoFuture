# ADR-003: Приоритетная очередность выделения сервисов

## Контекст

Монолит "GoFuture" содержит 9 доменных компонентов (Booking, Driver, Pricing, Payments, Payouts, Notification, Geography, Analytics, Fraud), которые необходимо выделить в независимые микросервисы. Все компоненты обращаются к общей PostgreSQL-базе напрямую, многие вызывают друг друга синхронно. Декомпозиция должна быть поэтапной с минимизацией простоя и рисков.

Текущие проблемы, влияющие на приоритеты:

- Пиковые нагрузки (>100K поездок) вызывают каскадные отказы - нужно изолировать критический путь.
- Единая БД PostgreSQL - узкое место масштабирования.
- Сборка монолита занимает >30 мин - независимые сервисы ускорят CI/CD.
- MTTR >4 часов - нужна наблюдаемость и изоляция отказов.

## Требования

Определить последовательность выделения сервисов из монолита с обоснованием логики, учитывая:

- Бизнес-импакт (какие сервисы дают наибольшую ценность при выделении).
- Техническую сложность (степень связанности с другими доменами).
- Риск (финансовые и регуляторные последствия ошибок).
- Зависимости (какие сервисы нужны как prerequisites для других).

## Решение

Используем паттерн Strangler Fig - постепенное выделение сервисов из монолита с параллельной работой старого и нового кода через фасадный слой (API Gateway + Anti-Corruption Layer).

### Критерии оценки

Каждый сервис оценен по 4 критериям (1-5, где 5 - лучше):

| Критерий                  | Описание                                                                             |
| ------------------------- | ------------------------------------------------------------------------------------ |
| Бизнес-ценность выделения | Насколько независимый деплой и масштабирование этого сервиса ускоряют бизнес         |
| Простота выделения        | Насколько легко отделить от монолита (мало sync-зависимостей, четкие границы данных) |
| Низкий риск               | Насколько безопасно выделять (не финансовый, не критический путь)                    |
| Prerequisite-ценность     | Насколько выделение этого сервиса упрощает последующие выделения                     |

### Оценка сервисов

| Сервис       | Бизнес-ценность | Простота | Низкий риск | Prerequisite | Итого |
| ------------ | --------------- | -------- | ----------- | ------------ | ----- |
| Notification | 3               | 5        | 5           | 2            | 15    |
| Geography    | 4               | 4        | 4           | 5            | 17    |
| Pricing      | 5               | 4        | 4           | 4            | 17    |
| Fraud        | 3               | 4        | 3           | 3            | 13    |
| Analytics    | 3               | 3        | 5           | 1            | 12    |
| Driver       | 5               | 3        | 3           | 4            | 15    |
| Payments     | 5               | 2        | 1           | 2            | 10    |
| Booking      | 5               | 1        | 2           | 1            | 9     |

### План выделения по этапам

#### Этап 0: Подготовка инфраструктуры (Месяцы 1-2)

Цель: Создать фундамент для микросервисной архитектуры.

Задачи:

1. API Gateway - развернуть Kong/Envoy как единую точку входа. Все клиентские запросы проксируются через Gateway -> монолит. Это позволит перенаправлять трафик на новые сервисы без изменения клиентов.
2. Identity Service - выделить аутентификацию/авторизацию из монолита. Все сервисы будут использовать единый OAuth 2.0 + JWT.
3. Event Bus (Kafka) - развернуть Kafka для доменных событий. Монолит начинает публиковать события параллельно с текущей логикой.
4. Observability stack - настроить трейсинг (Jaeger/Tempo), обновить Prometheus/Grafana, внедрить structured logging.
5. Kubernetes - развернуть кластер, настроить CI/CD пайплайн для деплоя микросервисов (замена Docker Compose).
6. Service Mesh (Istio/Linkerd) - mTLS, circuit breaker, retry, observability на уровне инфраструктуры.

Без этих компонентов невозможно безопасно выделять сервисы - не будет маршрутизации трафика, аутентификации, асинхронного взаимодействия и наблюдаемости.

Промежуточный результат: Монолит работает как раньше, но за API Gateway. Все запросы проходят через Gateway, логируются, трассируются.

#### Этап 1: Notification Service (Месяцы 2-3)

Почему первый:

- Минимальная связанность - не вызывает другие доменные сервисы синхронно, только получает задачи из очереди.
- Уже асинхронный - в монолите уведомления отправляются через Celery-задачи; переход почти не меняет архитектуру.
- Низкий риск - ошибка в уведомлениях не приводит к потере денег или поездок.
- Хорошая "тренировка" - команда отрабатывает процесс выделения сервиса на наименее рискованном домене.

Что делаем:

1. Создаем Notification Service с собственной БД (notification_templates, device_tokens, preferences, history).
2. Сервис подписывается на события из Kafka (BookingCreated, TripCompleted и т.д.) и отправляет push через FCM/APNS/Huawei.
3. В монолите заменяем прямые вызовы notification_domain на публикацию событий в Kafka.
4. Параллельная работа: на переходном этапе и монолит, и новый сервис обрабатывают уведомления; после стабилизации монолит отключается.

Данные для миграции: ~5 таблиц, низкий объем, нет связей с финансовыми данными.

#### Этап 2: Geography Service (Месяцы 3-5)

Почему второй:

- Высокая prerequisite-ценность - Geography нужен Booking, Pricing и Driver для sync-вызовов. Выделение Geography позволяет этим сервисам сразу вызывать его через API, а не через монолит.
- Четкие границы - владеет геоданными, Elasticsearch-индексом, кешем в Redis; почти не читает чужие таблицы.
- Внешняя зависимость - интеграция с Яндекс Картами изолируется в одном сервисе, что упрощает смену провайдера для новых регионов.
- Масштабирование - геопоиск - одна из самых нагруженных операций; независимое масштабирование даст существенный выигрыш.

Что делаем:

1. Создаем Geography Service с Elasticsearch, Redis-кешем и интеграцией с Яндекс Картами.
2. Предоставляем gRPC/REST API для геокодирования, маршрутизации, ETA, геопоиска водителей.
3. В монолите вызовы geography_domain заменяем на вызовы нового сервиса через HTTP/gRPC.
4. Anti-Corruption Layer в монолите преобразует старые внутренние вызовы в API-вызовы нового сервиса.

Данные для миграции: Elasticsearch-индекс (пересоздается), geo_zones из PostgreSQL (~3 таблицы), Redis-кеш (cold start допустим).

#### Этап 3: Pricing Service (Месяцы 5-6)

Почему третий:

- Высокая бизнес-ценность - динамическое ценообразование - ключевое конкурентное преимущество; независимый деплой позволяет быстро экспериментировать с тарифами.
- Четкие границы - владеет pricing_rules, tariffs, surge_zones; данные других доменов получает через API Geography (уже выделен).
- Независимое масштабирование - surge pricing создает пиковую нагрузку, которую нужно масштабировать отдельно.

Что делаем:

1. Создаем Pricing Service с собственной БД и Redis-кешем для ценовых правил.
2. API: расчет стоимости поездки, surge-коэффициент, расчет заработка водителя.
3. Вызывает Geography Service для расстояния/времени маршрута.
4. Подписывается на Kafka-события для обновления ML-модели спроса.

Данные для миграции: ~5 таблиц (pricing_rules, tariffs, surge_zones, surge_history, driver_earnings_rules). Средний объем, нет перекрестных FK.

#### Этап 4: Fraud Service (Месяцы 6-7)

Почему четвертый:

- Необходим перед выделением Payments - Payments вызывает Fraud синхронно для проверки транзакций. Выделить Fraud отдельно до Payments проще, чем выделять оба одновременно.
- Четкие границы - владеет собственными ML-моделями и правилами; читает данные других доменов, но только для обогащения (можно заменить на события).
- Средний риск - ложноположительные срабатывания (false positive) раздражают пользователей, но false negative - это финансовые потери.

Что делаем:

1. Создаем Fraud Service с собственной БД для правил, scores и incidents.
2. API: sync-проверка заказа/транзакции, оценка риска.
3. Подписывается на Kafka-события (BookingCreated, PaymentCompleted) для обновления ML-модели.
4. Монолит заменяет вызовы fraud_domain на вызовы API нового сервиса.

Данные для миграции: ~5 таблиц (fraud_rules, fraud_scores, fraud_incidents, blacklists). Средний объем.

#### Этап 5: Driver Service (Месяцы 7-9)

Почему пятый:

- Высокая бизнес-ценность - "умное" перераспределение водителей - ключевая функция; независимый деплой ускоряет итерации.
- Средняя сложность - зависит от Geography (уже выделен), но на него зависят Booking и Payments (Payouts).
- Подготовка к выделению Booking - после выделения Driver, у Booking остается меньше прямых зависимостей в монолите.

Что делаем:

1. Создаем Driver Service с собственной БД для профилей, статусов, рейтингов, документов.
2. API: CRUD профиля, управление статусом, matching (назначение водителя на заказ с учетом "умного" перераспределения).
3. Вызывает Geography Service для позиций водителей.
4. Публикует события DriverAssigned, DriverLocationUpdated в Kafka.

Данные для миграции: ~6 таблиц (drivers, driver_profiles, driver_statuses, driver_ratings, driver_documents, driver_vehicles). Большой объем - у платформы сотни тысяч водителей.

#### Этап 6: Payments Service (Месяцы 9-11)

Почему шестой:

- Высочайший риск - финансовые данные, PCI DSS, регуляторные требования. Нужен максимальный опыт команды (накоплен за предыдущие выделения).
- Зависимости готовы - Fraud (уже выделен) и Driver (уже выделен) доступны через API.
- Большой бизнес-импакт - независимый Payment Service позволяет интегрировать локальные платежные шлюзы для новых регионов без изменения остальной системы.

Что делаем:

1. Создаем Payments Service (включая Payouts) с отдельной PCI DSS-compliant БД.
2. API: платежи, возвраты, привязка карт, инициация выплат, баланс водителя.
3. Интеграция с Яндекс Пэй, API банка и будущими локальными шлюзами.
4. Строгий аудит всех операций; шифрование карточных данных.

Данные для миграции: ~8 таблиц (payments, payment_methods, transactions, refunds, payouts, payout_schedules, driver_balances, audit_log). Большой объем, высокая критичность - нужна верификация каждой записи.

#### Этап 7: Booking Service + Analytics Service (Месяцы 11-12)

Почему последние:

- Booking - самый связанный домен: вызывает Pricing, Geography, Driver, Fraud. Все эти сервисы должны быть выделены и стабилизированы до выделения Booking.
- Analytics - подписчик всех событий. Его можно выделить только когда все сервисы публикуют события в Kafka.
- К этому моменту монолит содержит только Booking-логику + Analytics ETL, что упрощает финальное выделение.

Booking Service:

1. Создаем Booking Service с собственной БД (bookings, trip_statuses, trip_history).
2. Вызывает через API: Pricing, Geography, Driver, Fraud (все уже выделены).
3. Публикует события жизненного цикла поездки в Kafka.
4. Монолит полностью выводится из эксплуатации после миграции.

Analytics Service:

1. Создаем Analytics Service на базе Spark/Flink + ClickHouse.
2. Подписывается на все Kafka-топики для event-driven аналитики (замена прямых SQL-запросов к общей БД).
3. Предоставляет API для отчетов и прогнозов.
4. Интеграция с DataLens для BI.

Данные для миграции:

- Booking: ~4 таблицы, очень большой объем (миллионы поездок).
- Analytics: ClickHouse уже существует; нужно переключить источник данных с ETL из PostgreSQL на Kafka-стримы.

## Альтернативы

1. Начать с Booking (core domain first) - максимальный бизнес-импакт сразу, но максимальный риск из-за высокой связанности. Ошибки на первом сервисе подорвут доверие к процессу миграции.
2. Big bang (все сразу) - минимальный период сосуществования монолита и микросервисов, но неприемлемый риск и требует заморозки разработки.
3. Выделять Analytics раньше - снимает нагрузку аналитических запросов с PostgreSQL, но Analytics зависит от событий всех сервисов; пока сервисы в монолите, Analytics придется читать и из Kafka, и из общей БД.

## Компромиссы

- 12 месяцев - это реалистичный срок. В случае задержек на ранних этапах последние могут сдвинуться.
- Параллельная работа монолита и микросервисов (Strangler Fig) увеличивает операционную сложность на 12 месяцев, но исключает big bang риски.
- Notification первым - не дает максимального бизнес-импакта, но минимизирует риск первого выделения и позволяет команде отладить процесс.
- Booking последним - бизнес ждет масштабирования Booking, но его раннее выделение слишком рискованно. Параллельно можно оптимизировать Booking в монолите (кеширование, оптимизация запросов).
