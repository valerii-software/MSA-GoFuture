# ADR-004: Механизмы обеспечения обратной совместимости

## Контекст

При поэтапном переходе от монолита к микросервисам (Strangler Fig) система будет находиться в гибридном состоянии на протяжении ~12 месяцев: часть функциональности обслуживается монолитом, часть - новыми сервисами. Клиентские приложения (iOS, Android, Huawei, Web) не могут быть обновлены одномоментно - в рабочем режиме всегда присутствуют несколько версий мобильных приложений. Корпоративные партнеры интегрированы через API и не могут мгновенно адаптироваться к изменениям.

Необходимо обеспечить:

- Бесшовный переход для клиентских приложений без принудительного обновления.
- Совместимость API для корпоративных партнеров.
- Корректную работу системы в гибридном режиме (часть сервисов - в монолите, часть - выделена).
- Возможность отката при обнаружении проблем.

## Требования

Разработать механизмы обратной совместимости, обеспечивающие непрерывную работу платформы на всех этапах миграции.

## Решение

### 1. API Gateway как фасадный слой (Strangler Fig Facade)

API Gateway является ключевым компонентом для обеспечения обратной совместимости. Он выступает единой точкой входа для всех клиентов и скрывает внутреннюю топологию сервисов.

Механизм маршрутизации:

Общая диаграмма механизмов обратной совместимости: [diagrams/strangler-fig-routing.puml](diagrams/strangler-fig-routing.puml)

- Все клиентские приложения обращаются к единому URL API Gateway.
- Gateway маршрутизирует запросы на основе конфигурации:
  - До выделения сервиса: `/api/v1/notifications/*` -> Монолит
  - После выделения: `/api/v1/notifications/*` -> Notification Service
- Переключение происходит на уровне конфигурации Gateway без изменения клиентов.

Канареечное переключение:

1. 100% трафика -> Монолит (до выделения).
2. 5% трафика -> Новый сервис, 95% -> Монолит (canary).
3. 25% -> 50% -> 100% трафика -> Новый сервис (постепенный rollout).
4. Мониторинг на каждом шаге: error rate, latency (p50/p95/p99), бизнес-метрики.
5. Автоматический откат при превышении порогов (error rate >1%, latency p99 >2x baseline).

### 2. Версионирование API

Стратегия: URL-based versioning с контрактом обратной совместимости.

```
/api/v1/bookings   - текущая версия (поддерживается монолитом и новым сервисом)
/api/v2/bookings   - новая версия (только новый сервис)
```

Правила версионирования:

| Тип изменения                              | Пример                             | Новая версия API?                   |
| ------------------------------------------ | ---------------------------------- | ----------------------------------- |
| Добавление нового поля в ответ             | `+ "eta_minutes": 5`               | Нет (additive, backward-compatible) |
| Добавление опционального параметра запроса | `+ ?currency=USD`                  | Нет                                 |
| Удаление поля из ответа                    | `- "old_field"`                    | Да (breaking change)                |
| Изменение типа поля                        | `"price": "100"` -> `"price": 100` | Да                                  |
| Изменение семантики поля                   | `"status"` меняет набор значений   | Да                                  |
| Новый обязательный параметр                | `+ "region_id" (required)`         | Да                                  |

Политика поддержки версий:

- Одновременно поддерживаются не более 2 версий API (текущая + предыдущая).
- Предыдущая версия (N-1) поддерживается минимум 6 месяцев после выпуска новой.
- Клиенты уведомляются через `Deprecation` HTTP-header и документацию.
- Для корпоративных партнеров - индивидуальные сроки миграции по договоренности.

Deprecation workflow:

```
1. Выпуск v2 -> v1 помечается deprecated (header: Deprecation: true)
2. +3 месяца -> уведомление партнеров о предстоящем sunset
3. +6 месяцев -> v1 отключается (или возвращает 410 Gone)
```

### 3. Anti-Corruption Layer (ACL)

ACL - адаптерный слой, который преобразует контракты между монолитом и новыми сервисами.

Где применяется: ACL живет на стороне вызывающего. Монолит вызывает ACL-адаптер, который преобразует старый формат в новый API нового сервиса, и наоборот. Потоки показаны на диаграмме [diagrams/strangler-fig-routing.puml](diagrams/strangler-fig-routing.puml).

Пример: выделение Geography Service

До выделения монолит вызывал geography_domain напрямую:

```python
# Внутренний вызов в монолите
route = geography_domain.calculate_route(from_lat, from_lng, to_lat, to_lng)
distance: float = route.distance_km
```

ACL в монолите заменяет вызов на HTTP-запрос к новому сервису:

```python
# ACL-адаптер в монолите
class GeographyServiceAdapter:
    def calculate_route(self, from_lat, from_lng, to_lat, to_lng) -> RouteResult:
        response = http_client.get(
            f"{GEOGRAPHY_SERVICE_URL}/api/v1/routes",
            params={"from": f"{from_lat},{from_lng}", "to": f"{to_lat},{to_lng}"}
        )
        # Преобразование нового формата API к старому внутреннему формату
        return RouteResult(
            distance_km=response["distance"]["value"] / 1000,  # новый API возвращает метры
            duration_sec=response["duration"]["value"]
        )
```

Принципы ACL:

- ACL живет на стороне вызывающего (монолита или нового сервиса).
- ACL - временный слой, удаляется после полной миграции вызывающей стороны.
- ACL не содержит бизнес-логики, только трансформацию данных.
- ACL покрывается интеграционными тестами (contract tests).

### 4. Механизм двойной записи (Dual Write) и Feature Flags

На переходном этапе, когда сервис уже выделен, но монолит еще не отключен:

Dual Write (для критических данных):

- Новый сервис является source of truth для своих данных.
- Изменения синхронизируются в монолитную БД через события (Kafka -> consumer в монолите), чтобы компоненты монолита, еще не выделенные, могли читать актуальные данные.
- После выделения всех зависимых компонентов синхронизация отключается.

Feature Flags:

```yaml
# Пример конфигурации feature flags
feature_flags:
  use_geography_service:
    enabled: true
    rollout_percentage: 100 # Постепенный rollout
    fallback: "monolith" # При ошибке - fallback на монолит

  use_pricing_service:
    enabled: true
    rollout_percentage: 25 # Канареечный этап
    fallback: "monolith"
```

- Feature flag определяет, куда направляется запрос: в монолит или в новый сервис.
- Позволяет мгновенный откат (переключение флага) без деплоя.
- Флаги управляются централизованно (LaunchDarkly / Unleash / собственный сервис).

### 5. Contract Testing

Контрактные тесты гарантируют, что API сервисов не ломают ожидания потребителей.

Стратегия: Consumer-Driven Contract Testing (Pact)

- Consumer описывает ожидаемый формат ответа (contract).
- Producer верифицирует, что его API соответствует всем контрактам потребителей.
- Контракты хранятся в Pact Broker и проверяются в CI/CD-пайплайне.
- Breaking change блокирует деплой, если хотя бы один consumer не обновлен.

Пример контрактного теста:

```
Consumer: Booking (в монолите)
Provider: Geography Service

Interaction: "запрос маршрута"
  Request:
    GET /api/v1/routes?from=55.75,37.62&to=55.73,37.59
  Expected Response:
    Status: 200
    Body:
      distance.value: integer (>0)
      distance.unit: "meters"
      duration.value: integer (>0)
      duration.unit: "seconds"
```

### 6. Schema Registry для событий

Для асинхронного взаимодействия через Kafka используется Schema Registry (Confluent Schema Registry / Apicurio).

Правила эволюции схем событий:

| Изменение                     | Совместимость       | Допустимо?                              |
| ----------------------------- | ------------------- | --------------------------------------- |
| Добавление опционального поля | Backward-compatible | Да                                      |
| Удаление опционального поля   | Forward-compatible  | Да (с предупреждением)                  |
| Добавление обязательного поля | Breaking            | Нет - создать новую версию события      |
| Переименование поля           | Breaking            | Нет - добавить новое, deprecated старое |

Формат: Avro или Protobuf (строгая типизация, компактность, поддержка эволюции схем).

Версионирование событий:

```
topic: booking.events.v1   # BookingCreated v1
topic: booking.events.v2   # BookingCreated v2 (после breaking change)
```

Consumer-ы подписываются на нужную версию топика. Старые версии поддерживаются до миграции всех consumer-ов.

### 7. Database Views для переходного периода

Пока не все сервисы выделены, компоненты монолита могут нуждаться в данных, которые "переехали" в новый сервис.

Подход: materialized views + CDC (Change Data Capture)

Поток данных: Новый сервис (новая БД) -> CDC (Debezium) -> Kafka -> Consumer -> Materialized View (монолитная БД). Детали показаны на диаграмме [diagrams/strangler-fig-routing.puml](diagrams/strangler-fig-routing.puml).

- Новый сервис пишет в свою БД (source of truth).
- Debezium публикует изменения в Kafka.
- Consumer в монолите обновляет materialized view в старой PostgreSQL.
- Компоненты монолита читают из view как раньше (без изменения кода).
- View удаляется после выделения всех зависимых компонентов.

Пример для Geography Service:

```sql
-- Materialized view в монолитной PostgreSQL
-- Обновляется через CDC из БД Geography Service
CREATE MATERIALIZED VIEW geo_zones_view AS
SELECT zone_id, name, polygon, surge_multiplier, updated_at
FROM geo_zones_sync;  -- таблица, обновляемая CDC-consumer-ом

-- Компоненты монолита продолжают читать как раньше
SELECT * FROM geo_zones_view WHERE ST_Contains(polygon, point);
```

### 8. Стратегия отката (Rollback)

| Уровень              | Механизм                                                                 | Время отката |
| -------------------- | ------------------------------------------------------------------------ | ------------ |
| API Gateway          | Переключение маршрутизации на монолит                                    | < 1 мин      |
| Feature Flag         | Переключение флага на "monolith"                                         | < 1 мин      |
| Canary deployment    | Автоматический откат при превышении error rate                           | < 2 мин      |
| Data sync            | Остановка CDC, монолит снова является source of truth                    | < 5 мин      |
| Полный откат сервиса | Отключение сервиса, возврат трафика на монолит, пересинхронизация данных | < 30 мин     |

Критерии автоматического отката:

- Error rate нового сервиса > 1% (при baseline < 0,1%).
- Latency p99 > 2x от baseline.
- Бизнес-метрика (conversion rate, количество успешных бронирований) снизилась > 5%.

## Альтернативы

1. Принудительное обновление клиентов - проще технически, но неприемлемо для бизнеса: пользователи теряют доступ к приложению, корпоративные партнеры не могут обновиться мгновенно.
2. GraphQL вместо REST versioning - гибкое эволюционирование API без версий, но требует значительных инвестиций в инфраструктуру и обучение команд; не оправдано на этапе миграции.
3. Event versioning через полиморфизм (одно событие с union-типами) - меньше топиков, но сложнее обработка и десериализация на стороне consumer.

## Компромиссы

- Dual Write создает сложность синхронизации и потенциальную рассогласованность. Митигация: CDC с at-least-once гарантией + idempotent consumer. Это временная мера.
- Два брокера (Kafka + RabbitMQ) на переходном этапе увеличивают операционные затраты. Митигация: RabbitMQ остается только для Celery-задач внутри сервисов, Kafka - для межсервисных событий.
- Contract testing замедляет CI/CD (дополнительные тесты), но предотвращает runtime-ошибки несовместимости, которые значительно дороже.
- Materialized views через CDC вносят задержку (секунды) в обновление данных в монолите. Для real-time операций (геопоиск) используется прямой API-вызов, а не view.
- Поддержка двух версий API удваивает объем тестирования для задетых эндпоинтов, но необходима для плавного перехода партнеров.
