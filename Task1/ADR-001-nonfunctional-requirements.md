# ADR-001: Нефункциональные требования

## Контекст

Компания "GoFuture" осуществляет переход от монолитной архитектуры к микросервисной. Монолит не справляется с масштабированием при пиковых нагрузках (>100K активных поездок), время разработки новых функций превышает 6 месяцев, а MTTR - более 4 часов. Компания выходит на рынки Юго-Восточной Азии и Южной Америки, что требует мультирегионального развертывания и соответствия локальным регуляторным требованиям.

Нефункциональные требования определяют архитектурные ограничения, которые должны быть заложены в целевую микросервисную архитектуру.

## Требования

Сформулировать измеримые нефункциональные требования для целевой архитектуры, которые обеспечат достижение бизнес-целей компании.

## Решение

### 1. Производительность

| Метрика                            | Целевое значение | Обоснование                                              |
| ---------------------------------- | ---------------- | -------------------------------------------------------- |
| Конкурентные поездки               | >= 500 000       | Ключевое бизнес-требование                               |
| Латентность API (p95), чтение      | <= 200 мс        | UX мобильных приложений                                  |
| Латентность API (p95), запись      | <= 500 мс        | UX при создании заказа                                   |
| Расчет динамической цены           | <= 100 мс        | Real-time ценообразование                                |
| Подбор водителя (matching)         | <= 3 с           | Конкурентный SLA                                         |
| Геопоиск ближайших водителей       | <= 500 мс        | Real-time обновление карты                               |
| Обработка платежа                  | <= 2 с           | UX завершения поездки                                    |
| Пропускная способность API Gateway | >= 50 000 RPS    | Расчет: 500K поездок \* ~100 API-вызовов/поездка / 1000с |

### 2. Доступность

| Метрика                                                  | Целевое значение                  | Обоснование                     |
| -------------------------------------------------------- | --------------------------------- | ------------------------------- |
| Критические сервисы (Booking, Driver, Payments, Pricing) | >= 99,95% (<= 22 мин простоя/мес) | Бизнес-требование               |
| Важные сервисы (Geography, Fraud, Notification)          | >= 99,9% (<= 44 мин простоя/мес)  | Влияют на UX, но не блокируют   |
| Вспомогательные сервисы (Analytics)                      | >= 99,5%                          | Допустима задержка в отчетах    |
| Развертывание                                            | Zero-downtime deployment          | Непрерывность бизнеса           |
| Аварийное переключение между регионами                   | <= 30 с (автоматическое)          | Мультирегиональная устойчивость |

### 3. Масштабируемость

| Метрика                           | Целевое значение                        | Обоснование                              |
| --------------------------------- | --------------------------------------- | ---------------------------------------- |
| Горизонтальное масштабирование    | >= 3 географических регионов            | Бизнес-требование                        |
| Автоскейлинг                      | По CPU/RPS/Custom metrics               | Замена ручного масштабирования           |
| Время масштабирования (scale-out) | <= 60 с                                 | Обработка пиковых нагрузок               |
| Независимое масштабирование       | Каждый сервис - отдельно                | Оптимизация ресурсов (цель: -40% затрат) |
| Масштабирование БД                | Read replicas + шардирование по региону | Локализация данных                       |

### 4. Надежность

| Метрика                        | Целевое значение                                        | Обоснование                         |
| ------------------------------ | ------------------------------------------------------- | ----------------------------------- |
| MTTR                           | <= 15 мин (финал), <= 1 ч (промежуточно)                | Снижение с текущих 4+ часов         |
| MTBF                           | >= 720 ч для критических сервисов                       | Стабильность платформы              |
| RPO (Recovery Point Objective) | <= 1 мин для финансовых данных                          | Минимизация потери данных           |
| RTO (Recovery Time Objective)  | <= 5 мин для критических сервисов                       | Быстрое восстановление              |
| Изоляция отказов               | Отказ одного сервиса не вызывает каскадный отказ        | Устранение текущей проблемы         |
| Circuit breaker                | Срабатывание при >50% ошибок за 10 с                    | Защита от каскадных отказов         |
| Retry с exponential backoff    | 3 попытки, макс. задержка 30 с                          | Устойчивость к транзиентным ошибкам |
| Graceful degradation           | Бронирование работает при отказе Analytics/Notification | Приоритет критического пути         |

### 5. Безопасность

| Метрика                               | Целевое значение                                    | Обоснование                      |
| ------------------------------------- | --------------------------------------------------- | -------------------------------- |
| Аутентификация                        | OAuth 2.0 + JWT, централизованный Identity Provider | Единый вход для всех клиентов    |
| Авторизация                           | RBAC с поддержкой мультитенантности                 | Партнерская модель               |
| Шифрование данных в транзите          | TLS 1.3 (mTLS между сервисами)                      | Защита межсервисного трафика     |
| Шифрование данных в покое             | AES-256                                             | Защита персональных данных       |
| PCI DSS                               | Уровень 1 для платежного сервиса                    | Обработка карточных данных       |
| Соответствие регуляторным требованиям | Локальные законы о персональных данных (SEA, SA)    | Экспансия на новые рынки         |
| Rate limiting                         | Настраиваемый per-tenant, per-endpoint              | Защита от DDoS и злоупотреблений |
| Аудит                                 | Логирование всех операций с PII и финансами         | Регуляторные требования          |

### 6. Наблюдаемость

| Метрика                              | Целевое значение                        | Обоснование                          |
| ------------------------------------ | --------------------------------------- | ------------------------------------ |
| Покрытие метрик критических сервисов | 100%                                    | Промежуточная метрика                |
| Distributed tracing                  | 100% для критического пути бронирования | Диагностика латентности              |
| Sampling rate (остальные запросы)    | >= 10%                                  | Баланс объема данных и наблюдаемости |
| Централизованное логирование         | Все сервисы, structured logging (JSON)  | Быстрый поиск проблем                |
| Время обнаружения инцидента (MTTD)   | <= 2 мин                                | Автоматический алертинг              |
| Retention метрик                     | >= 30 дней (hot), >= 1 год (cold)       | Trend analysis и compliance          |
| Retention логов                      | >= 14 дней (hot), >= 90 дней (cold)     | Debugging и аудит                    |

### 7. Развертывание и разработка

| Метрика                             | Целевое значение                     | Обоснование                     |
| ----------------------------------- | ------------------------------------ | ------------------------------- |
| Время сборки одного сервиса         | <= 5 мин                             | Снижение с 30+ мин для монолита |
| Время выхода новой функции на рынок | <= 2 недели                          | Бизнес-требование               |
| Независимость деплоя                | Каждый сервис деплоится независимо   | Ускорение разработки            |
| Стратегия деплоя                    | Canary / Blue-Green                  | Zero-downtime                   |
| Откат (rollback)                    | <= 5 мин, автоматический по метрикам | Снижение риска релизов          |
| Feature flags                       | Поддержка на уровне платформы        | Постепенный rollout             |

### 8. Данные и согласованность

| Метрика                 | Целевое значение                       | Обоснование                         |
| ----------------------- | -------------------------------------- | ----------------------------------- |
| Модель согласованности  | Eventual consistency (Saga pattern)    | Распределенные транзакции           |
| Строгая согласованность | Только внутри одного сервиса           | Упрощение архитектуры               |
| Суверенитет данных      | Данные хранятся в регионе пользователя | Регуляторные требования             |
| Database per service    | Каждый сервис владеет своими данными   | Loose coupling                      |
| Потеря событий          | <= 0,001% (at-least-once delivery)     | Надежность event-driven архитектуры |

## Альтернативы

1. Менее строгие SLA (99,9% вместо 99,95%) - снижает затраты на инфраструктуру, но не соответствует бизнес-требованию.
2. Strong consistency через distributed transactions (2PC) - проще для разработчиков, но создает связанность между сервисами и снижает доступность.
3. Единая БД для всех сервисов (shared database) - проще миграция, но не обеспечивает независимое масштабирование и деплой сервисов.

## Компромиссы

- Eventual consistency усложняет разработку (нужны Saga, компенсирующие транзакции), но обеспечивает независимость сервисов и высокую доступность.
- mTLS между сервисами добавляет ~1-2 мс латентности к каждому вызову, но необходим для безопасности.
- 99,95% доступности требует значительных инвестиций в инфраструктуру (мультирегиональность, автоматический failover), но является бизнес-требованием.
- Database per service увеличивает операционную сложность (больше баз данных для поддержки), но обеспечивает независимость команд и сервисов.
- Distributed tracing создает дополнительную нагрузку на систему, поэтому для некритического пути используется sampling.
