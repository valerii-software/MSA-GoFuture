# ADR-005: План миграции данных

## Контекст

Монолит "GoFuture" использует единую PostgreSQL RDS как основную базу данных для всех доменов. Все доменные компоненты обращаются к общим таблицам напрямую через ORM (Django) и прямые SQL-запросы. Существуют перекрестные зависимости: Analytics читает все таблицы, Geography обращается к drivers и zones, Fraud читает payments и bookings.

При переходе к микросервисной архитектуре каждый сервис должен владеть своими данными (Database per Service). Это означает:

- Миграцию данных из общей PostgreSQL в отдельные базы данных сервисов.
- Работу с большими объемами данных (миллионы поездок, сотни тысяч водителей, миллионы транзакций).
- Обеспечение нулевого простоя и согласованности данных во время миграции.
- Удаление перекрестных зависимостей (прямые SQL-запросы к чужим таблицам).

## Требования

Разработать план миграции данных для каждого сервиса с учетом большого объема данных и нулевого простоя.

## Решение

### Общая стратегия миграции

Для каждого сервиса применяется четырехфазный подход:

Диаграмма: [diagrams/migration-phases.puml](diagrams/migration-phases.puml)

Инструменты:

- Debezium - Change Data Capture для PostgreSQL (логическая репликация).
- Apache Kafka - транспорт для CDC-событий.
- pg_dump / COPY - bulk-миграция исторических данных.
- Flyway / Liquibase - управление схемами новых БД.

### Миграция по сервисам

#### 1. Notification Service (Этап 1, месяцы 2-3)

Таблицы-источники (монолитная PostgreSQL):
| Таблица | Примерный объем | Описание |
|---------|----------------|----------|
| notification_templates | ~200 записей | Шаблоны уведомлений |
| device_tokens | ~2M записей | Токены устройств пользователей |
| notification_preferences | ~1M записей | Предпочтения пользователей |
| notification_log | ~50M записей | История отправленных уведомлений |

Целевая БД: PostgreSQL (отдельный инстанс).

Стратегия:

1. Bulk-миграция справочников (templates, preferences) через pg_dump -> pg_restore. Объем мал - минуты.
2. Bulk-миграция device_tokens через `COPY ... TO` / `COPY ... FROM` с конвертацией. ~2M записей - до 10 мин.
3. notification_log - миграция hot-данных (последние 30 дней, ~5M записей) через COPY. Исторические данные (>30 дней) - через фоновый batch job в течение нескольких дней. Старые данные доступны через монолит до завершения.
4. CDC (Debezium) - включается после bulk-миграции для синхронизации изменений в реальном времени (новые device_tokens, обновленные preferences).
5. Cutover - переключение через API Gateway. Монолит перестает писать в notification-таблицы. CDC отключается.

FK-зависимости:

- device_tokens.user_id -> users (будущий Identity Service). На переходном этапе user_id хранится как значение (не FK), валидация через API Identity Service.
- notification_preferences.user_id -> аналогично.

Откат: Восстановление маршрутизации на монолит через API Gateway (<1 мин). Данные, записанные в новую БД, синхронизируются обратно через reverse-CDC или ручной скрипт.

#### 2. Geography Service (Этап 2, месяцы 3-5)

Таблицы-источники:
| Таблица | Примерный объем | Описание |
|---------|----------------|----------|
| geo_zones | ~5K записей | Географические зоны |
| geo_regions | ~500 записей | Регионы (страны, города) |
| routes_cache | ~10M записей | Кешированные маршруты |

Дополнительные хранилища:
| Хранилище | Объем | Описание |
|-----------|-------|----------|
| Elasticsearch (driver_locations) | ~500K документов | Real-time позиции водителей |
| Redis (geo_cache) | ~1GB | Кеш геоданных |

Целевая БД: PostgreSQL (PostGIS) + Elasticsearch + Redis.

Стратегия:

1. geo_zones, geo_regions - bulk-миграция через pg_dump. Объем мал, завершается за минуты.
2. routes_cache - миграция hot-данных (последние 7 дней). Старые записи не мигрируются (кеш будет прогрет заново).
3. Elasticsearch - индекс пересоздается с нуля; данные о позициях водителей поступают в реальном времени от Driver -> Geography API. Нет необходимости мигрировать - данные эфемерные.
4. Redis - cold start. Кеш прогревается по мере обработки запросов. Допустимо временное увеличение латентности (first-call penalty).
5. CDC для geo_zones - синхронизация изменений зон до полного cutover.
6. Materialized view в монолитной БД для компонентов, которые еще читают geo_zones (см. ADR-004).

FK-зависимости:

- geo_zones ссылается на geo_regions - оба мигрируют вместе, FK сохраняются.
- Другие домены (Pricing, Driver) читали geo_zones напрямую -> после миграции обращаются через API Geography Service.

Откат: Маршрутизация через API Gateway. Elasticsearch пересоздается за минуты.

#### 3. Pricing Service (Этап 3, месяцы 5-6)

Таблицы-источники:
| Таблица | Примерный объем | Описание |
|---------|----------------|----------|
| pricing_rules | ~1K записей | Правила ценообразования |
| tariffs | ~5K записей | Тарифные планы |
| surge_zones | ~10K записей | Зоны surge-ценообразования |
| surge_history | ~20M записей | История surge-коэффициентов |
| driver_earnings_rules | ~500 записей | Правила расчета заработка |

Целевая БД: PostgreSQL + Redis (кеш ценовых правил).

Стратегия:

1. Справочники (pricing_rules, tariffs, driver_earnings_rules) - bulk через pg_dump. Минуты.
2. surge_zones - bulk через COPY, CDC для изменений.
3. surge_history - миграция hot-данных (последние 90 дней, ~5M записей) через COPY. Архив - batch job в фоне.
4. Redis - cold start, кеш прогревается при старте сервиса (preload pricing_rules и tariffs).
5. CDC для справочников - синхронизация до cutover.

FK-зависимости:

- surge_zones ранее ссылался на geo_zones -> после миграции хранит zone_id как значение, валидация через API Geography Service.
- Нет FK на другие домены.

Откат: API Gateway routing + Feature Flag.

#### 4. Fraud Service (Этап 4, месяцы 6-7)

Таблицы-источники:
| Таблица | Примерный объем | Описание |
|---------|----------------|----------|
| fraud_rules | ~500 записей | Правила обнаружения фрода |
| fraud_scores | ~5M записей | Скоры пользователей |
| fraud_incidents | ~100K записей | Инциденты фрода |
| blacklists | ~50K записей | Черные списки |
| ml_fraud_models | ~10 записей (+ модели в S3) | ML-модели |

Целевая БД: PostgreSQL.

Стратегия:

1. fraud_rules, blacklists, ml_fraud_models - bulk через pg_dump. Секунды.
2. fraud_scores - bulk через COPY (~5M записей, ~30 мин). CDC для новых/обновленных scores.
3. fraud_incidents - bulk через COPY (~100K записей, секунды). CDC для новых инцидентов.
4. ML-модели - копирование артефактов из текущего хранилища в S3/Object Storage нового сервиса.

FK-зависимости:

- fraud_scores.user_id -> users - хранится как значение, валидация через Identity Service API.
- fraud_incidents ранее ссылался на bookings, payments - хранится booking_id/payment_id как значение, обогащение через API (или Kafka-события).

Откат: API Gateway + Feature Flag. Fraud scores восстанавливаются из монолитной БД.

#### 5. Driver Service (Этап 5, месяцы 7-9)

Таблицы-источники:
| Таблица | Примерный объем | Описание |
|---------|----------------|----------|
| drivers | ~500K записей | Основные данные водителей |
| driver_profiles | ~500K записей | Профили водителей |
| driver_statuses | ~500K записей | Текущие статусы |
| driver_ratings | ~10M записей | Оценки и рейтинги |
| driver_documents | ~2M записей | Документы (права, страховки) |
| driver_vehicles | ~600K записей | Транспортные средства |

Целевая БД: PostgreSQL.

Стратегия:

1. drivers, driver_profiles, driver_vehicles - bulk через COPY. ~500K записей - 10-15 мин.
2. driver_statuses - bulk не нужен. Новый сервис начинает собирать статусы сам. Текущие статусы инициализируются как "offline", реальные - обновляются при следующем подключении водителя.
3. driver_ratings - bulk миграция hot-данных (последние 12 месяцев, ~8M записей) через batch COPY. ~1 час в фоне.
4. driver_documents - bulk через COPY + миграция файлов в Object Storage нового сервиса. Фоновая задача - несколько часов.
5. CDC для drivers, driver_profiles - синхронизация изменений до cutover.
6. Materialized view в монолитной БД для Booking (если еще не выделен) и Payments.

FK-зависимости (критические):

- Payouts (в монолите) читает drivers для выплат -> materialized view + планируемое выделение в Payments Service.
- Booking (в монолите) назначает driver_id -> materialized view до выделения Booking.
- Geography обновляет driver_locations -> после миграции Driver публикует позиции через API Geography Service.

Откат: API Gateway + Feature Flag + reverse-CDC для обратной синхронизации.

#### 6. Payments Service (Этап 6, месяцы 9-11)

Таблицы-источники:
| Таблица | Примерный объем | Описание |
|---------|----------------|----------|
| payments | ~50M записей | Платежи от пассажиров |
| payment_methods | ~3M записей | Платежные методы |
| transactions | ~100M записей | Транзакции (дебет/кредит) |
| refunds | ~1M записей | Возвраты |
| payouts | ~20M записей | Выплаты водителям |
| payout_schedules | ~500K записей | Расписания выплат |
| driver_balances | ~500K записей | Балансы водителей |
| audit_log | ~200M записей | Аудит-лог всех операций |

Целевая БД: PostgreSQL (PCI DSS-compliant, отдельная сеть, шифрование).

Стратегия - наиболее консервативная из-за финансовой критичности:

1. Подготовка (2 недели):
   - Создание PCI DSS-compliant инфраструктуры (изолированная сеть, шифрование, аудит).
   - Развертывание новой PostgreSQL с шифрованием at rest (AES-256).
   - Настройка Debezium CDC для всех финансовых таблиц.

2. Bulk-миграция по фазам:
   - Фаза A: payment_methods, payout_schedules, driver_balances - малый объем, через COPY. Верификация: count + checksum.
   - Фаза B: payments, refunds - ~51M записей. Batch COPY по 1M записей за раз, с верификацией каждого batch (row count + sum(amount)).
   - Фаза C: transactions - ~100M записей. Batch COPY по 1M, верификация sum(amount) за каждый день. Ожидаемое время: 4-6 часов.
   - Фаза D: payouts - ~20M записей. Batch COPY, верификация sum(amount).
   - Фаза E: audit_log - ~200M записей. Фоновая миграция в течение нескольких дней. Не блокирует cutover (аудит-лог не влияет на бизнес-логику).

3. Верификация:
   - Row count - количество записей совпадает (±допуск на новые записи).
   - Financial checksum - sum(amount) по каждой таблице за каждый день совпадает до копейки.
   - Random sampling - случайная выборка 10K записей, побайтовое сравнение.
   - Reconciliation report - автоматический отчет о расхождениях.

4. CDC-синхронизация:
   - Debezium захватывает все изменения в монолитной БД после начала bulk-миграции.
   - Consumer применяет изменения к новой БД, исключая дубликаты (idempotent upsert).
   - Лаг CDC < 5 секунд.

5. Cutover (переключение):
   - Канареечный режим: 5% трафика -> новый Payments Service, 95% -> монолит.
   - Мониторинг: error rate, latency, финансовые метрики (sum транзакций за час).
   - Постепенный rollout: 5% -> 25% -> 50% -> 100%.
   - Dual-write на переходном этапе: новый сервис пишет в свою БД, CDC синхронизирует обратно в монолит (для Booking, который еще в монолите).

6. Post-cutover:
   - Мониторинг 72 часа после полного переключения.
   - Финансовая сверка (reconciliation) каждый час.
   - Отключение CDC после подтверждения стабильности.

FK-зависимости:

- payments.booking_id -> bookings (еще в монолите) - хранится как значение, обогащение через Kafka-событие TripCompleted.
- payments.user_id -> users - через Identity Service API.
- payouts.driver_id -> drivers - через Driver Service API.

Откат: API Gateway + Feature Flag. Финансовые данные синхронизированы через CDC в обоих направлениях. Максимальная потеря: транзакции за время лага CDC (< 5 сек).

#### 7. Booking Service (Этап 7, месяцы 11-12)

Таблицы-источники:
| Таблица | Примерный объем | Описание |
|---------|----------------|----------|
| bookings | ~100M записей | Все бронирования |
| trip_statuses | ~100M записей | Статусы поездок (история) |
| trip_history | ~100M записей | Детали завершенных поездок |
| trip_routes | ~50M записей | Маршруты поездок |

Целевая БД: PostgreSQL (с шардированием по region_id для мультирегионального развертывания).

Стратегия:

1. Шардирование по region_id:
   - Каждый регион получает свой шард: `bookings_sea` (Юго-Восточная Азия), `bookings_sa` (Южная Америка), `bookings_ru` (Россия/СНГ).
   - Маршрутизация на уровне приложения (по region_id из JWT-токена или запроса).

2. Bulk-миграция:
   - Партиционирование по дате создания (месячные партиции).
   - Миграция hot-данных (последние 6 месяцев, ~30M записей) - batch COPY по 1M записей. ~3-4 часа.
   - Архивные данные (>6 месяцев) - фоновая миграция в течение нескольких дней.

3. CDC для bookings, trip_statuses - синхронизация новых записей до cutover.

4. Cutover:
   - К этому моменту все зависимые сервисы (Pricing, Geography, Driver, Fraud, Payments) уже выделены и стабильны.
   - Booking Service вызывает их через API.
   - Канареечный rollout: 5% -> 25% -> 50% -> 100%.

5. Деcomission монолита:
   - После переключения 100% трафика - монолит остается в read-only режиме 2 недели (safety net).
   - Полное отключение монолита после верификации.

FK-зависимости:

- bookings.driver_id -> drivers - через Driver Service API.
- bookings.payment_id -> payments - через Payments Service API (или Kafka-событие).
- Все FK заменены на идентификаторы + API-вызовы.

Откат: Последний сервис - откат означает возврат к монолиту. CDC обеспечивает синхронизацию данных в обоих направлениях.

#### 8. Analytics Service (Этап 7, месяцы 11-12)

Источники данных:
| Источник | Описание |
|----------|----------|
| ClickHouse (существующий) | Аналитические данные, уже агрегированные |
| PostgreSQL (все таблицы) | Прямые SQL-запросы для отчетов |
| Kafka-события | Новый источник - события всех сервисов |

Целевая архитектура: Kafka -> Spark/Flink -> ClickHouse -> DataLens.

Стратегия:

1. Переключение источника данных:
   - Вместо прямых SQL-запросов к монолитной PostgreSQL, Analytics переходит на Kafka-стримы.
   - К этапу 7 все сервисы публикуют доменные события в Kafka -> Analytics подписывается на все топики.

2. ClickHouse:
   - Существующий ClickHouse сохраняется (исторические данные).
   - Новые данные поступают из Kafka через Kafka Engine в ClickHouse (или через Flink).
   - Нет необходимости мигрировать данные - только переключить источник.

3. ETL-пайплайны:
   - Существующие Spark/Flink джобы адаптируются для чтения из Kafka вместо PostgreSQL.
   - Постепенное переключение: сначала новые джобы читают из Kafka, старые продолжают работать.
   - После стабилизации - отключение старых ETL-пайплайнов.

4. ML-пайплайны:
   - ML-модели (прогнозирование спроса) переобучаются на данных из Kafka-стримов.
   - Feature store обновляется из Kafka вместо SQL-запросов.

Откат: Возврат ETL-пайплайнов на чтение из PostgreSQL (конфигурационное изменение).

### Общие правила миграции

1. Никогда не удалять данные из монолитной БД до полного подтверждения стабильности нового сервиса (минимум 2 недели после cutover).
2. Все миграции выполняются вне пиковых часов (ночь/раннее утро по основному региону).
3. Мониторинг lag CDC - алерт при лаге > 30 секунд.
4. Автоматические тесты верификации запускаются после каждого batch и перед cutover.
5. Данные с PII (персональные данные) мигрируются с шифрованием в транзите (TLS) и в покое (AES-256).
6. Foreign keys между сервисами заменяются на идентификаторы (UUID) + API-вызовы или Kafka-события для обогащения.

## Альтернативы

1. Shared database (без миграции данных) - все сервисы продолжают использовать общую PostgreSQL. Проще в реализации, но полностью блокирует независимое масштабирование и деплой сервисов. Не соответствует целевой архитектуре.
2. Big bang миграция (все данные за один раз) - короче переходный период, но требует длительного простоя (часы) для синхронизации и верификации. Неприемлемо для бизнеса.
3. CQRS с Event Sourcing - полный переход на событийную модель данных. Элегантно, но требует переписывания всей бизнес-логики. Слишком дорого и рискованно для первой итерации.

## Компромиссы

- CDC через Debezium добавляет нагрузку на монолитную PostgreSQL (логическая репликация использует WAL). Митигация: мониторинг replication lag, настройка publication для конкретных таблиц.
- Eventual consistency между сервисами - данные могут рассогласовываться на секунды. Для финансовых операций - дополнительная reconciliation. Для остальных - приемлемо.
- Materialized views в монолитной БД - временная мера, создающая дополнительную нагрузку. Удаляются после выделения всех зависимых компонентов.
- Дублирование данных (dual-write период) - увеличивает затраты на хранение, но обеспечивает безопасный откат.
- Отказ от FK между сервисами снижает referential integrity - компенсируется проверками в коде и reconciliation-скриптами.
