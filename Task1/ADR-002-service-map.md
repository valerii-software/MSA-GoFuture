# ADR-002: Карта сервисов и доменные границы

## Контекст

В монолите "GoFuture" все доменные компоненты (Booking, Driver, Pricing, Payments, Payouts, Notification, Geography, Analytics, Fraud) находятся в единой кодовой базе Django и обращаются к общей PostgreSQL-базе. Между доменами существуют прямые синхронные вызовы и перекрестный доступ к таблицам других доменов (например, Analytics читает все таблицы, Geography обращается к таблицам drivers и zones, Fraud читает payments и bookings).

Организационная структура включает 8 продуктовых команд, каждая из которых отвечает за свой домен. Согласно закону Конвея, архитектура сервисов должна соответствовать структуре команд.

## Требования

Определить границы доменных сервисов, их ответственности, владеемые данные и контракты взаимодействия - с учетом минимизации связанности (coupling) и максимизации связности (cohesion).

## Решение

### Принципы определения границ

1. Один домен - один сервис - одна команда.
2. Database per service - каждый сервис владеет своими данными и предоставляет доступ только через API.
3. Синхронные вызовы - только для запросов на критическом пути (бронирование поездки).
4. Асинхронные события - для уведомлений, аналитики, побочных эффектов.
5. API Gateway - единая точка входа для клиентских приложений.

### Карта сервисов

#### 1. Booking Service

| Параметр                    | Значение                                                                                                        |
| --------------------------- | --------------------------------------------------------------------------------------------------------------- |
| Команда                     | Booking                                                                                                         |
| Ответственность             | Управление жизненным циклом поездки: создание, назначение водителя, отслеживание статуса, завершение            |
| Владеемые данные            | bookings, trip_statuses, trip_history                                                                           |
| Входящие API                | Создание заказа, получение статуса, отмена, история поездок                                                     |
| Исходящие синхронные вызовы | Pricing (расчет цены), Geography (маршрут и ETA), Driver (поиск и назначение водителя), Fraud (проверка заказа) |
| Публикуемые события         | BookingCreated, BookingAccepted, TripStarted, TripCompleted, BookingCancelled                                   |
| Подписка на события         | DriverLocationUpdated (обновление ETA), PaymentCompleted (подтверждение оплаты)                                 |

#### 2. Driver Service

| Параметр                    | Значение                                                                                                  |
| --------------------------- | --------------------------------------------------------------------------------------------------------- |
| Команда                     | Driver                                                                                                    |
| Ответственность             | Управление профилями водителей, статусами (онлайн/офлайн/на поездке), "умное" перераспределение, рейтинги |
| Владеемые данные            | drivers, driver_profiles, driver_statuses, driver_ratings, driver_documents                               |
| Входящие API                | Регистрация, обновление профиля, смена статуса, получение назначения, подтверждение поездки               |
| Исходящие синхронные вызовы | Geography (текущая позиция для matching)                                                                  |
| Публикуемые события         | DriverOnline, DriverOffline, DriverLocationUpdated, DriverAssigned, DriverRatingUpdated                   |
| Подписка на события         | BookingCreated (для matching), TripCompleted (обновление статуса)                                         |

#### 3. Pricing Service

| Параметр                    | Значение                                                                                           |
| --------------------------- | -------------------------------------------------------------------------------------------------- |
| Команда                     | Pricing                                                                                            |
| Ответственность             | Динамическое ценообразование, расчет тарифов, surge pricing, расчет заработка водителя             |
| Владеемые данные            | pricing_rules, tariffs, surge_zones, surge_history, driver_earnings_rules                          |
| Входящие API                | Расчет стоимости поездки, получение текущего surge-коэффициента, расчет заработка водителя         |
| Исходящие синхронные вызовы | Geography (расстояние и время маршрута для расчета)                                                |
| Публикуемые события         | PriceCalculated, SurgeUpdated                                                                      |
| Подписка на события         | BookingCreated/TripCompleted (для ML-модели спроса), DriverLocationUpdated (предложение в регионе) |

#### 4. Payments Service

| Параметр                    | Значение                                                                                                         |
| --------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| Команда                     | Payments                                                                                                         |
| Ответственность             | Прием платежей от пассажиров, интеграция с платежными шлюзами, управление платежными методами, выплаты водителям |
| Владеемые данные            | payments, payment_methods, transactions, refunds, payouts, payout_schedules, driver_balances                     |
| Входящие API                | Инициация платежа, возврат, привязка карты, история транзакций, инициация выплаты, баланс водителя               |
| Исходящие синхронные вызовы | Яндекс Пэй (процессинг), API Банка (выплаты), Fraud (проверка транзакции)                                        |
| Публикуемые события         | PaymentCompleted, PaymentFailed, RefundProcessed, PayoutCompleted, PayoutFailed                                  |
| Подписка на события         | TripCompleted (инициация платежа), BookingCancelled (возврат)                                                    |

Примечание: Payouts (выплаты водителям) объединены с Payments в один сервис, так как оба домена работают с финансовыми транзакциями, разделяют требования PCI DSS, и команда Payments уже владеет обеими функциями. Это снижает операционную сложность управления финансовыми данными.

#### 5. Notification Service

| Параметр                    | Значение                                                                                                              |
| --------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| Команда                     | Notification                                                                                                          |
| Ответственность             | Отправка push-уведомлений, SMS, email; управление шаблонами и предпочтениями пользователей                            |
| Владеемые данные            | notification_templates, notification_preferences, device_tokens, notification_history                                 |
| Входящие API                | Отправка уведомления, управление предпочтениями, история уведомлений                                                  |
| Исходящие синхронные вызовы | FCM, APNS, Huawei Push Kit                                                                                            |
| Публикуемые события         | NotificationSent, NotificationFailed                                                                                  |
| Подписка на события         | BookingCreated, BookingAccepted, TripStarted, TripCompleted, PaymentCompleted, PayoutCompleted (триггеры уведомлений) |

#### 6. Geography Service

| Параметр                    | Значение                                                                                             |
| --------------------------- | ---------------------------------------------------------------------------------------------------- |
| Команда                     | Geography                                                                                            |
| Ответственность             | Геокодирование, построение маршрутов, расчет ETA, управление геозонами, геоиндекс водителей          |
| Владеемые данные            | geo_zones, geo_regions, driver_locations (real-time в Redis/Elasticsearch), routes_cache             |
| Входящие API                | Геокодирование адреса, построение маршрута, расчет ETA, поиск водителей в радиусе, получение геозоны |
| Исходящие синхронные вызовы | Яндекс Карты (геоданные и маршруты)                                                                  |
| Публикуемые события         | DriverLocationUpdated (после обновления позиции), GeozoneEntered, GeozoneExited                      |
| Подписка на события         | DriverOnline/DriverOffline (добавление/удаление из геоиндекса)                                       |

#### 7. Analytics Service

| Параметр                    | Значение                                                                          |
| --------------------------- | --------------------------------------------------------------------------------- |
| Команда                     | Analytics                                                                         |
| Ответственность             | Сбор и агрегация бизнес-событий, ML-пайплайны (прогнозирование спроса), BI-отчеты |
| Владеемые данные            | analytics_events, aggregated_metrics, ml_models, reports (ClickHouse)             |
| Входящие API                | Запрос отчетов, запрос прогнозов спроса                                           |
| Исходящие синхронные вызовы | - (не вызывает другие сервисы синхронно)                                          |
| Публикуемые события         | DemandForecastUpdated (для Pricing)                                               |
| Подписка на события         | Все бизнес-события от всех сервисов (event consumer)                              |

#### 8. Fraud Service

| Параметр                    | Значение                                                                                                              |
| --------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| Команда                     | Fraud                                                                                                                 |
| Ответственность             | Обнаружение мошенничества в реальном времени, проверка транзакций, оценка риска, блокировка подозрительной активности |
| Владеемые данные            | fraud_rules, fraud_scores, fraud_incidents, blacklists, ml_fraud_models                                               |
| Входящие API                | Проверка заказа на фрод, проверка транзакции, оценка риска пользователя                                               |
| Исходящие синхронные вызовы | - (принимает решение на основе собственных данных и ML-модели)                                                        |
| Публикуемые события         | FraudDetected, UserBlocked, RiskScoreUpdated                                                                          |
| Подписка на события         | BookingCreated, PaymentCompleted (для обновления ML-модели и обогащения данных)                                       |

### Инфраструктурные сервисы

#### API Gateway

| Параметр        | Значение                                                                                 |
| --------------- | ---------------------------------------------------------------------------------------- |
| Команда         | Platform                                                                                 |
| Ответственность | Маршрутизация запросов, аутентификация, rate limiting, API versioning, агрегация ответов |
| Технология      | Kong / Envoy                                                                             |

#### Identity Service

| Параметр         | Значение                                                                                    |
| ---------------- | ------------------------------------------------------------------------------------------- |
| Команда          | Platform                                                                                    |
| Ответственность  | Аутентификация, авторизация (OAuth 2.0 + JWT), управление пользователями, мультитенантность |
| Владеемые данные | users, roles, permissions, sessions, tenants                                                |

#### Event Bus

| Параметр        | Значение                                           |
| --------------- | -------------------------------------------------- |
| Команда         | Platform                                           |
| Ответственность | Маршрутизация доменных событий между сервисами     |
| Технология      | Apache Kafka (замена RabbitMQ для event streaming) |

Примечание: RabbitMQ остается для task queue (внутри сервисов, где используется Celery). Kafka используется как шина доменных событий между сервисами - он лучше подходит для event sourcing, имеет гарантию порядка в рамках партиции, поддерживает replay событий и масштабируется горизонтально.

| Booking       | -       | +      | +       | -        | -            | +         | -         | +     |
| Driver        | -       | -      | -       | -        | -            | +         | -         | -     |
| Pricing       | -       | -      | -       | -        | -            | +         | -         | -     |
| Payments      | -       | -      | -       | -        | -            | -         | -         | +     |
| Notification  | -       | -      | -       | -        | -            | -         | -         | -     |
| Geography     | -       | -      | -       | -        | -            | -         | -         | -     |
| Analytics     | -       | -      | -       | -        | -            | -         | -         | -     |
| Fraud         | -       | -      | -       | -        | -            | -         | -         | -     |

Notification, Analytics - подписчики событий, не делают синхронных вызовов к другим доменным сервисам.  
Payments инициирует платеж по событию TripCompleted (async), но проверяет фрод синхронно.

## Альтернативы

1. Объединить Booking + Driver - снижает количество sync-вызовов, но создает слишком большой сервис с двумя командами и разными циклами релизов.
2. Выделить Payouts в отдельный сервис - возможно в будущем при росте команды, но сейчас увеличивает операционную сложность без явной выгоды, так как Payouts и Payments разделяют финансовые данные и требования compliance.
3. Оставить RabbitMQ как единственный брокер - проще в эксплуатации, но не поддерживает replay событий и плохо масштабируется для event streaming между множеством потребителей.

## Компромиссы

- Sync-вызовы Booking -> Pricing/Geography/Driver/Fraud создают runtime-зависимость. Митигация: circuit breaker, timeout, fallback (например, использование кешированной цены при недоступности Pricing).
- Объединение Payments + Payouts в один сервис нарушает принцип "один домен - один сервис", но оправдано общими данными и compliance-требованиями. При росте команды можно разделить.
- Два брокера (Kafka + RabbitMQ) увеличивают операционную сложность, но каждый оптимален для своей задачи: Kafka для event streaming, RabbitMQ для task queue.
- Analytics подписывается на все события - потенциально высокая нагрузка. Митигация: отдельная consumer group, собственный ClickHouse, батчевая обработка.
