# ADR-010: Глобальное развертывание и защита платформы

## Контекст

GoFuture выходит на рынки Юго-Восточной Азии (SEA) и Южной Америки (SA). Требуется обеспечить 99,99% доступности (<= 52 мин простоя/год) и низкую латентность (p95 <= 200 мс) для пользователей во всех регионах. Текущее развертывание в одном регионе не обеспечивает ни необходимой отказоустойчивости (single region - single point of failure), ни приемлемой латентности для удаленных рынков (RTT Европа -> SEA ~250 мс, Европа -> SA ~200 мс).

Дополнительно необходимо:

- Соответствие локальным регуляторным требованиям.
- Защита платформы от внешних угроз (DDoS, injection, bot-атаки).
- Развертывание в более чем 3 географических регионах.

## Решение

### 1. Выбор регионов развертывания

| Регион          | Локация             | Роль                   | Обоснование                                                                                                                                                                                  |
| --------------- | ------------------- | ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| SGP (Singapore) | Сингапур            | Primary SEA            | Сетевой хаб SEA: RTT < 30 мс до Таиланда, Вьетнама, Малайзии, Филиппин. Зрелая инфраструктура всех облачных провайдеров. Юрисдикция PDPA - базовый уровень защиты данных для SEA             |
| JKT (Jakarta)   | Джакарта, Индонезия | Secondary SEA          | Крупнейший рынок ride-hailing в SEA (270M населения). PP 71/2019 требует локального хранения данных для публичных электронных систем. RTT SGP <-> JKT ~15 мс - эффективная пара для failover |
| GRU (São Paulo) | Сан-Паулу, Бразилия | Primary SA             | Крупнейший рынок SA (210M населения). RTT < 40 мс до Аргентины, Чили, Колумбии. LGPD требует контроля над обработкой персональных данных. Доступность всех крупных облачных провайдеров      |
| FRA (Frankfurt) | Франкфурт, Германия | Control Plane / DR Hub | Географический центр между SEA и SA (RTT ~120 мс до обоих). Хранение глобальных данных (Identity, Reference). DR-площадка для GRU. GDPR-совместимая юрисдикция для европейских партнеров     |

Почему именно 4 региона:

- SGP + JKT - покрытие SEA с failover-парой. Индонезия - слишком крупный рынок для обслуживания из Сингапура (RTT ~15 мс приемлемо, но data residency требует локального хранения).
- GRU - единственный регион в SA с зрелой облачной инфраструктурой.
- FRA - control plane для глобальных сервисов (Identity, Reference Data, Analytics), DR для GRU, минимизация RTT для управления.

Калькуляция доступности:

Один регион с 3 AZ: ~99,95%. Два региона с автоматическим failover: 1 - (0,0005)\*\*2 = 99,99975%. Таким образом, пара SGP <-> JKT и пара GRU <-> FRA обеспечивают целевые 99,99%.

### 2. Схема репликации данных между регионами

Диаграмма: [diagrams/data-replication.puml](diagrams/data-replication.puml)

#### Классификация данных по стратегии репликации

| Категория                 | Данные                                         | Где Primary         | Репликация         | Метод                                    | Задержка |
| ------------------------- | ---------------------------------------------- | ------------------- | ------------------ | ---------------------------------------- | -------- |
| Глобальные (Global)       | users, roles, permissions, sessions            | FRA                 | FRA -> все регионы | PostgreSQL streaming replication (async) | < 500 мс |
| Справочные (Reference)    | pricing_rules, tariffs, geo_zones, fraud_rules | FRA                 | FRA -> все регионы | CDC (Debezium) -> Kafka -> consumer      | < 2 с    |
| Операционные (Regional)   | bookings, trips, drivers, payments, payouts    | Регион пользователя | НЕ реплицируются   | -                                        | -        |
| Кеши (Cache)              | driver_locations, surge, sessions              | Регион              | НЕ реплицируются   | Локальный Redis/ES                       | -        |
| Аналитические (Analytics) | бизнес-события всех регионов                   | FRA                 | Все регионы -> FRA | Kafka MirrorMaker 2                      | < 5 с    |

#### Принцип: Regional Ownership

Каждый регион полностью автономен для обработки поездок своих пользователей:

- Операционные данные (bookings, drivers, payments) остаются в регионе и не реплицируются между регионами. Пассажир и водитель всегда принадлежат одному региону (определяется при регистрации или по текущей локации).
- Глобальные данные (Identity) реплицируются из FRA во все регионы как read-only реплики. Регистрация и изменение профиля - через FRA (async, < 500 мс). Аутентификация - по локальной реплике (< 5 мс).
- Справочные данные обновляются централизованно (FRA) и распространяются через CDC. Каждый регион хранит полную копию справочников.
- Аналитика - события из всех регионов агрегируются в FRA через Kafka MirrorMaker 2 для глобальных отчетов и ML-моделей.

#### Конфликты и разрешение

| Сценарий                                          | Решение                                                                                                     |
| ------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| Пользователь путешествует между регионами         | Первый заказ в новом регионе -> запрос к FRA Identity -> создание операционного профиля в локальном регионе |
| Одновременное обновление профиля из двух регионов | Last-write-wins на уровне FRA Identity (через timestamp)                                                    |
| Недоступность FRA для записи                      | Регионы продолжают работать на локальных репликах (read-only); обновления профилей откладываются в очередь  |

### 3. Геомаршрутизация пользователей

Диаграмма: [diagrams/geo-routing.puml](diagrams/geo-routing.puml)

#### Механизм маршрутизации

```
Пользователь -> api.gofuture.com -> GeoDNS (Cloudflare) -> SGP/JKT/GRU/FRA API Gateway
```

Уровень 1: DNS-based Geo-routing (Cloudflare DNS)

- Домен `api.gofuture.com` разрешается в IP ближайшего региона.
- Метод: Latency-based routing - Cloudflare измеряет RTT от resolver-a клиента до каждого региона и возвращает ближайший.
- TTL: 30 секунд - баланс между скоростью failover и нагрузкой на DNS.
- Geo-фильтры: IP-адреса из Индонезии -> JKT, из SEA -> SGP, из SA -> GRU, остальные -> FRA.

Уровень 2: Anycast + Global Accelerator

- Каждый регион имеет Anycast IP через Cloudflare.
- При отказе региона трафик автоматически перенаправляется на следующий по latency.
- Переключение: < 30 секунд (TTL DNS + propagation).

Уровень 3: Client-side Region Affinity

- После первой аутентификации JWT-токен содержит `region_id` (SGP/JKT/GRU/FRA).
- Мобильное приложение кеширует IP региона и отправляет запросы напрямую.
- При ошибке 5xx или timeout -> fallback на GeoDNS (повторный DNS-запрос).

Уровень 4: API Gateway Region-aware Routing

- Если запрос попал в "чужой" регион (например, SGP-пользователь -> GRU API Gateway), Gateway возвращает `301 Redirect` с заголовком `X-Region: sgp` и URL правильного региона.
- Критические запросы (создание заказа) проксируются в правильный регион (а не редиректятся) для минимизации латентности.

### 4. Аварийное переключение (Failover)

Диаграмма: [diagrams/failover.puml](diagrams/failover.puml)

#### Failover-пары регионов

| Primary | Failover | RTT между ними | Сценарий                              |
| ------- | -------- | -------------- | ------------------------------------- |
| SGP     | JKT      | ~15 мс         | SEA-трафик переключается на JKT       |
| JKT     | SGP      | ~15 мс         | Indonesia-трафик переключается на SGP |
| GRU     | FRA      | ~120 мс        | SA-трафик переключается на FRA        |
| FRA     | SGP      | ~160 мс        | EU-трафик переключается на SGP        |

#### Health Checks

- Global Health Monitor (Cloudflare Health Checks): HTTP GET `/health` каждые 10 секунд из 5 точек мира.
- Критерий unhealthy: 3 последовательных неуспешных проверки (30 секунд).
- Критерий healthy: 2 последовательных успешных проверки (20 секунд).
- Endpoint `/health` проверяет: API Gateway up, >=1 instance каждого критического сервиса, PostgreSQL primary доступен, Kafka >=2 ISR.

#### Таблица сценариев отказа

| #   | Сценарий отказа                     | Точка отказа         | Обнаружение                 | Механизм переключения                                             | RTO    | RPO     |
| --- | ----------------------------------- | -------------------- | --------------------------- | ----------------------------------------------------------------- | ------ | ------- |
| 1   | Отказ pod/контейнера                | Service instance     | K8s liveness probe (10s)    | K8s перезапуск + HPA scale-up                                     | < 15 с | 0       |
| 2   | Отказ ноды K8s                      | Worker node          | K8s node controller (40s)   | Reschedule pods на другие ноды, Pod anti-affinity по AZ           | < 60 с | 0       |
| 3   | Отказ Availability Zone             | AZ                   | Regional health check       | K8s reschedule, pods распределены по 3 AZ (anti-affinity)         | < 30 с | 0       |
| 4   | Отказ региона (полный)              | Весь регион          | Global Health Monitor (30s) | GeoDNS failover на парный регион, HPA scale-up в failover-регионе | < 60 с | < 1 мин |
| 5   | Отказ PostgreSQL primary            | DB primary           | Patroni heartbeat (10s)     | Автоматический promote replica -> new primary (Patroni)           | < 30 с | < 1 с   |
| 6   | Отказ Kafka broker                  | Kafka broker         | ISR monitoring              | Automatic leader election (min.isr=2 из 3 brokers)                | < 10 с | 0       |
| 7   | Отказ внешнего сервиса (Яндекс Пэй) | External API         | Circuit breaker (10s)       | Fallback на альтернативный платежный шлюз; retry с backoff        | < 5 с  | 0       |
| 8   | Сетевой сбой между регионами        | Inter-region network | Latency/error мониторинг    | Автономная работа каждого региона; отложенная синхронизация       | < 10 с | < 5 мин |
| 9   | DDoS-атака                          | Ingress              | Cloudflare DDoS detection   | Auto-mitigation (rate limiting, GeoIP block), auto-scaling        | < 30 с | 0       |
| 10  | Деградация Debezium (CDC)           | CDC pipeline         | `outbox_pending_age > 30s`  | Переключение на Polling Publisher (fallback relay)                | < 60 с | < 30 с  |

#### Процедура регионального failover

1. Обнаружение (0-30 сек): Global Health Monitor фиксирует 3 неуспешных health check.
2. Решение (автоматическое): Cloudflare помечает регион как unhealthy.
3. Переключение DNS (30-60 сек): GeoDNS перестает возвращать IP неисправного региона; TTL=30s -> клиенты получают новый IP в течение 30 секунд.
4. Scale-up (60-120 сек): HPA в failover-регионе увеличивает количество pods для обработки дополнительной нагрузки.
5. Восстановление (ручное): После устранения неисправности - ручное включение региона обратно (canary 5% -> 25% -> 50% -> 100%).

#### Автономный режим региона

При потере связи с FRA (control plane) каждый регион продолжает работать автономно:

- Аутентификация - по локальной реплике Identity DB (read-only, JWT-валидация по публичному ключу).
- Справочные данные - по локальной копии (могут быть устаревшими на время недоступности FRA, приемлемо).
- Регистрация новых пользователей - откладывается в локальную очередь, синхронизируется после восстановления.
- Аналитика - события накапливаются в локальном Kafka, MirrorMaker доставит после восстановления.

### 5. Стратегия комплаенса

#### Регуляторные требования по регионам

| Регион | Регуляция                                            | Ключевые требования                                                                                          | Реализация                                                                                                                   |
| ------ | ---------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------- |
| SGP    | PDPA (Personal Data Protection Act)                  | Согласие на сбор/обработку PII; уведомление об утечке; назначение DPO                                        | Consent Management в Identity Service; Breach Notification pipeline; DPO per region                                          |
| JKT    | PP 71/2019 (GR on Electronic Systems) + PDP Law 2022 | Локальное хранение данных для публичных электронных систем; DPO; ограничение трансграничной передачи         | Все операционные данные JKT-пользователей хранятся только в JKT; трансграничная передача - только анонимизированные агрегаты |
| GRU    | LGPD (Lei Geral de Proteção de Dados)                | Правовое основание обработки; права субъектов (доступ, удаление, портабельность); DPO; уведомление об утечке | Data Subject Rights API; DPO per region; Audit log всех операций с PII                                                       |
| FRA    | GDPR (General Data Protection Regulation)            | Правовое основание; минимизация данных; права субъектов; Privacy by Design; DPO; уведомление в 72 часа       | GDPR-compliant Identity Service; Data Minimization в аналитике; DPIA для новых фич                                           |

#### Архитектурные решения для комплаенса

Data Residency (суверенитет данных):

- Операционные данные (bookings, trips, payments) никогда не покидают регион пользователя.
- PII (имя, телефон, email, платежные данные) хранятся только в регионе регистрации.
- Аналитические данные, передаваемые в FRA через MirrorMaker, анонимизируются перед отправкой:
  - `user_id` -> `hash(user_id + salt)` (псевдонимизация).
  - GPS-координаты -> агрегация до уровня геозоны (не точные координаты).
  - Платежные данные -> только суммы и статусы, без реквизитов карт.

Right to be Forgotten:

- API `DELETE /api/v1/users/{id}/data` запускает Saga удаления:
  1. Soft-delete в Identity DB (FRA).
  2. Команда на удаление операционных данных в регионе пользователя.
  3. Purge из всех кешей (Redis, ES).
  4. Отправка tombstone-событий в Kafka (для consumer-ов, хранящих данные).
  5. Аудит-запись о факте удаления (сохраняется для compliance).
- SLA: полное удаление в течение 30 дней.

Data Processing Agreement (DPA):

- Все облачные провайдеры подписывают DPA, соответствующий требованиям GDPR/LGPD.
- Субпроцессоры (Яндекс Пэй, провайдеры push-уведомлений) - отдельные DPA.
- Реестр субпроцессоров доступен пользователям.

Encryption:

| Слой                           | Метод                                 | Примечание               |
| ------------------------------ | ------------------------------------- | ------------------------ |
| В транзите (сервис <-> сервис) | mTLS (TLS 1.3)                        | Istio Service Mesh       |
| В транзите (клиент <-> сервер) | TLS 1.3                               | Cloudflare Edge          |
| В покое (БД)                   | AES-256 (Transparent Data Encryption) | Все регионы              |
| В покое (PCI DSS данные)       | AES-256 + field-level encryption      | Payments DB              |
| Ключи                          | HashiCorp Vault per region            | Ключи не покидают регион |

Аудит:

- Все операции с PII логируются в отдельный audit log (immutable, append-only).
- Retention: 5 лет (требование LGPD/PDPA для финансовых операций).
- Доступ к audit log - только через отдельную роль `compliance-auditor`.

### 6. Механизмы защиты платформы (WAF, Firewall, DDoS)

Обновленная C2-диаграмма: [C2-global-security.puml](C2-global-security.puml)

#### Новые компоненты безопасности

| Компонент             | Технология                   | Назначение                                                                                    | Размещение                   |
| --------------------- | ---------------------------- | --------------------------------------------------------------------------------------------- | ---------------------------- |
| DDoS Protection       | Cloudflare Shield            | Защита от L3/L4/L7 DDoS-атак; автоматическая митигация; GeoIP-фильтрация                      | Edge (перед всеми регионами) |
| WAF                   | Cloudflare WAF / ModSecurity | OWASP Top 10 (SQL injection, XSS, CSRF); кастомные правила для fraud-паттернов; Bot Detection | Edge + per-region            |
| Network Policy Engine | Calico / Cilium              | Zero-trust networking внутри K8s; микросегментация: сервис -> сервис ACL; L3/L4 правила       | Внутри каждого K8s-кластера  |
| Secret Manager        | HashiCorp Vault              | Управление секретами (API-ключи, DB credentials); автоматическая ротация; per-region инстанс  | В каждом регионе             |
| SIEM                  | Elastic SIEM / Wazuh         | Корреляция security-событий; обнаружение аномалий; incident response automation               | FRA (глобальный)             |

#### Обоснование выбора механизмов

DDoS Protection (Cloudflare Shield):

- GoFuture обрабатывает до 50K RPS на пике. DDoS-атака на ride-hailing платформу может парализовать транспорт в городе.
- Cloudflare обеспечивает Anycast-сеть в 310+ PoP, поглощая атаки до 100 Tbps на edge, не затрагивая origin.
- L7-защита: rate limiting per IP, per endpoint; challenge pages для подозрительного трафика.

WAF (Cloudflare WAF + ModSecurity):

- Два уровня: Cloudflare WAF на edge (быстрая фильтрация) + ModSecurity на API Gateway (глубокая инспекция).
- Managed rulesets: OWASP CRS (Core Rule Set) для стандартных атак.
- Custom rules: блокировка паттернов fraud-запросов (массовое создание заказов, перебор промокодов).
- Bot Detection: отсечение скраперов и автоматических клиентов; whitelist для партнерских API.

Network Policy Engine (Calico):

- Zero-trust: по умолчанию весь трафик между pods запрещен.
- Явные правила: Booking Service -> Pricing Service (port 8443) - разрешено; Notification Service -> Payments DB - запрещено.
- Логирование отклоненных соединений -> SIEM для анализа.

Secret Manager (HashiCorp Vault):

- Секреты (DB-пароли, API-ключи Яндекс Пэй, сертификаты mTLS) не хранятся в config-файлах или env-переменных.
- Автоматическая ротация DB-credentials каждые 24 часа.
- Per-region Vault инстанс - секреты не покидают регион (compliance).
- K8s integration через Vault Agent Injector.

SIEM (Elastic SIEM):

- Агрегация security-логов из всех регионов: WAF-события, Network Policy deny, failed auth attempts, Vault audit.
- Правила корреляции: 5+ failed login от одного IP за 1 мин -> auto-block; аномальный паттерн API-вызовов -> alert.
- Integration: Alertmanager -> SIEM -> PagerDuty (для security-инцидентов).

#### Поток трафика с учетом защиты

```
Пользователь -> Cloudflare Edge
-> GeoDNS
-> Regional API Gateway
-> Network Policy
-> Domain Service
-> Database
```

Каждый уровень - defense in depth. Компрометация одного слоя не дает доступа к данным.

## Альтернативы

1. Один регион + CDN - дешевле, но не обеспечивает 99,99% (single point of failure) и не решает data residency. CDN помогает со статикой, но не с API-запросами в real-time.
2. Active-Active все регионы с CRDTs - максимальная доступность, но CRDTs сложны для финансовых данных (payments, payouts) и создают трудноотлаживаемые конфликты.
3. AWS Global Accelerator вместо Cloudflare - лучшая интеграция с AWS, но vendor lock-in и отсутствие WAF/DDoS в комплекте (нужен отдельно AWS Shield + AWS WAF).
4. Self-hosted WAF (только ModSecurity) - полный контроль, но требует выделенной security-команды для обновления правил; Cloudflare managed rules снижают операционные затраты.
5. Три региона вместо четырех (без JKT) - дешевле, но не удовлетворяет требованиям data residency Индонезии и оставляет крупнейший SEA-рынок без локального развертывания.

## Компромиссы

- Regional Ownership означает, что пользователь "привязан" к региону. Путешествующий пользователь получит дополнительную латентность при первом заказе в новом регионе (~300 мс для миграции профиля). Митигация: pre-warm при обнаружении GPS в другом регионе.
- Async-репликация Identity DB допускает < 500 мс рассинхронизации. Изменение пароля в FRA может не дойти до SGP в течение ~500 мс. Митигация: критические auth-операции (смена пароля, блокировка) - synchronous через FRA с таймаутом 2 с.
- MirrorMaker 2 для аналитики добавляет задержку до 5 сек для глобальных отчетов. Для real-time дашбордов региональные Grafana показывают локальные данные; глобальный дашборд в FRA работает с ~5 сек задержкой.
- Cloudflare как единая точка для DDoS/WAF/DNS создает зависимость от одного вендора. Митигация: DNS failover через вторичный DNS-провайдер; возможность переключения на AWS CloudFront + Shield за 4 часа.
- 4 региона значительно увеличивают инфраструктурные затраты (~3x по сравнению с одним регионом). Митигация: FRA и JKT работают в "теплом" режиме (меньше ресурсов), автоскейлинг при failover.
- Per-region Vault усложняет управление секретами (синхронизация политик). Митигация: Vault Enterprise с namespace federation; Terraform для IaC.
- Zero-trust networking (Calico) замедляет коммуникацию между pods на ~0,5 мс (обработка правил). Для GoFuture с p95 < 200 мс - пренебрежимо.
